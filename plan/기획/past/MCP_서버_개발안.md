# 모바일인덱스 MCP 서버 — 개발 시 검토 사항

> MCP 서버를 만들 때 기획 단계에서 결정해야 하는 것들을 정리한 문서입니다.
> "코드를 어떻게 짜는가"가 아니라 "어떤 상품을 만들 것인가"에 초점을 맞춥니다.

<br>

## 1. 도구(Tool) 설계

MCP 서버의 핵심은 "도구"입니다. AI가 사용할 수 있는 기능 하나하나를 도구라고 부릅니다.
예: `search_app`, `app_usage`, `competitor_loyalty` 등

### 1-1. 도구를 어떻게 나눌 것인가

| 방식 | 설명 | 장단점 |
|------|------|--------|
| API 1개 = 도구 1개 | 기존 Insight API 엔드포인트를 그대로 도구로 매핑 | 단순하지만, 사용자가 여러 도구를 조합해야 하는 경우 AI가 알아서 판단해야 함 |
| 시나리오 단위 | "경쟁앱 종합 분석" 같은 복합 도구를 만들어 내부에서 여러 API를 호출 | 사용자 경험은 좋지만, 도구가 복잡해지고 유지보수 어려움 |
| 혼합 | 기본 도구(API 1:1) + 복합 도구(시나리오) 병행 | 현실적인 접근. 기본 도구로 시작하고, 사용 패턴 보고 복합 도구 추가 |

> 📌 현재 프로토타입은 API 1:1 방식(27개 도구). 베타 피드백 후 복합 도구 추가 검토 권장.

### 1-2. 도구 이름과 설명

AI는 도구의 이름과 설명(description)을 보고 어떤 도구를 쓸지 판단합니다.
이름과 설명이 모호하면 AI가 엉뚱한 도구를 선택하거나, 적절한 도구를 못 찾을 수 있습니다.

**좋은 예:**
```
이름: app_usage
설명: "특정 앱의 기간별 사용량을 분석합니다. 사용자 수, 사용 시간, 활성 기기, 신규 설치 추이."
```

**나쁜 예:**
```
이름: get_data
설명: "데이터를 가져옵니다"
```

> 📌 도구 설명은 AI가 읽는 것이므로, 사람이 아닌 AI 관점에서 명확하게 작성해야 합니다.
> 📌 한국어/영어 혼용 가능하지만, 일관성 유지 필요.

### 1-3. 파라미터 설계

각 도구에 어떤 입력값(파라미터)을 받을지 결정해야 합니다.

| 고려 사항 | 설명 |
|----------|------|
| 필수 vs 선택 | 앱 이름은 필수, OS 구분은 선택 등. 선택 파라미터가 많으면 AI가 알아서 판단 |
| 기본값 | 기간을 안 넣으면 최근 1개월로 자동 설정 등 |
| 파라미터 설명 | AI가 사용자 질문에서 어떤 값을 추출해야 하는지 알 수 있도록 설명 필요 |
| 날짜 형식 | "yyyymmdd" 같은 형식을 명시해야 AI가 올바른 형태로 변환 |

## 2. 응답 설계

MCP 서버가 AI에게 돌려주는 데이터의 형태를 결정해야 합니다.

### 2-1. 응답 포맷

| 방식 | 설명 | 적합한 경우 |
|------|------|-----------|
| JSON 원본 그대로 | API 응답을 가공 없이 전달 | 데이터가 단순하고, AI가 알아서 해석 가능할 때 |
| 요약 + 원본 | 핵심 수치를 텍스트로 요약하고, 상세 데이터는 JSON으로 첨부 | AI가 요약을 바로 활용하고, 필요 시 상세 데이터 참조 |
| 텍스트 변환 | JSON을 사람이 읽기 좋은 텍스트로 변환해서 전달 | AI가 리포트를 작성할 때 |

> 📌 현재 프로토타입은 JSON 원본 그대로 전달. AI가 잘 해석하고 있으나, 데이터가 많아지면 요약 방식 검토 필요.

### 2-2. 데이터가 없을 때

사용자가 존재하지 않는 앱이나 기간을 질의하면 어떻게 할 것인가.

| 상황 | 현재 동작 | 권장 동작 |
|------|----------|----------|
| 앱을 못 찾을 때 | API 에러 반환 | "해당 앱을 찾을 수 없습니다. 앱 이름을 다시 확인해주세요." |
| 데이터가 없는 기간 | 빈 배열 반환 | "해당 기간의 데이터가 없습니다. 조회 가능 기간: 2024년 1월~현재" |
| API 장애 | 에러 코드 반환 | "일시적으로 데이터를 조회할 수 없습니다. 잠시 후 다시 시도해주세요." |

> 📌 에러 메시지를 명확하게 작성하면, AI가 사용자에게 그대로 전달합니다.
> 📌 모호한 에러("Error 500")를 반환하면 AI가 할루시네이션으로 대체 답변을 만들 수 있습니다.

### 2-3. 할루시네이션 방지

MCP 서버가 직접 AI의 답변을 제어할 수는 없지만, 응답에 가이드를 포함시켜 유도할 수 있습니다.

**방법 1: 응답에 안내 문구 포함**
```json
{
  "data": [...],
  "notice": "이 데이터는 모바일인덱스 기준이며, 조회 시점의 데이터입니다. 데이터에 없는 내용은 추측하지 마세요."
}
```

**방법 2: 데이터 출처 명시**
```json
{
  "source": "모바일인덱스 Insight API",
  "date": "2026-03",
  "data": [...]
}
```

> 📌 AI는 출처가 명시된 데이터를 더 신뢰하고, 임의로 수치를 만들어내는 경향이 줄어듭니다.

## 3. 접근 제어

### 3-1. 무료/유료 구분

무료 사용자와 유료 사용자에게 다른 도구를 제공할 수 있습니다.

| 방식 | 설명 |
|------|------|
| 도구 자체를 숨기기 | 무료 사용자에게는 기본 도구만 노출, 심화 도구는 유료만 |
| 도구는 보이되 호출 차단 | 무료 사용자가 심화 도구 호출 시 "프리미엄 기능입니다" 메시지 반환 |
| 데이터 범위 제한 | 같은 도구지만 무료는 최근 1개월, 유료는 전체 기간 |

> 📌 "도구는 보이되 호출 차단" 방식이 업셀링에 효과적.
> 📌 사용자가 "이런 기능도 있구나"를 인지한 후 유료 전환 유도 가능.

### 3-2. 호출 제한 (Rate Limiting)

| 항목 | 설명 |
|------|------|
| 분당 호출 제한 | 1분에 n회 이상 호출 시 차단 (비정상 트래픽 방지) |
| 일일 호출 제한 | 무료 사용자 일 n회, 유료 사용자 무제한 |
| 동시 접속 제한 | 1개 API 키로 동시에 몇 개 세션까지 허용할지 |

> 📌 베타에서는 단순 키 목록 방식이라 세밀한 제한이 어려움.
> 📌 정식 출시 시 미들웨어 레벨에서 구현 필요.

## 4. 보안

### 4-1. API 토큰 관리

| 위험 | 설명 | 대응 |
|------|------|------|
| 토큰 노출 | MCP 서버 코드에 토큰이 하드코딩되면 유출 위험 | 환경변수로 관리 (현재 SSE 버전에 반영 완료) |
| 사용자 키 공유 | 1개 키를 여러 명이 공유해서 사용 | IP 기반 제한 또는 동시 세션 제한 |
| 키 탈취 | 사용자의 API 키가 유출될 경우 | 키 재발급 기능, 키 만료 정책 |

### 4-2. 데이터 보안

| 항목 | 설명 |
|------|------|
| HTTPS 필수 | AI 플랫폼 ↔ MCP 서버 간 통신 암호화 (AI 플랫폼이 필수 요구) |
| 로깅 | 누가 어떤 데이터를 조회했는지 기록 (감사 추적) |
| 데이터 캐싱 | MCP 서버에서 데이터를 캐싱할 경우, 캐시 만료 정책 필요 |

## 5. 성능

### 5-1. 응답 속도

AI 사용자는 대화형으로 질문하므로, 응답이 느리면 체감이 나쁩니다.

| 구간 | 목표 | 현재 |
|------|------|------|
| MCP 서버 → 모바일인덱스 API | 1초 이내 | 약 0.5~1초 |
| AI 플랫폼 → MCP 서버 | 2초 이내 | 로컬 테스트 기준 양호 |
| 전체 (사용자 질문 → 답변) | 5~10초 | AI 처리 시간 포함 |

> 📌 MCP 서버 자체는 빠르지만, AI가 여러 도구를 순차 호출하면 시간이 누적됩니다.
> 📌 복합 질의("경쟁앱 5개 비교") 시 도구를 5번 이상 호출할 수 있어 20~30초 소요 가능.

### 5-2. 데이터 크기

| 상황 | 문제 | 대응 |
|------|------|------|
| 대량 데이터 반환 | 순위 100개 앱 데이터를 한 번에 반환하면 AI 컨텍스트 윈도우 소모 | 기본 반환 개수 제한 (예: TOP 20), 사용자가 요청 시 확장 |
| 장기간 추이 데이터 | 12개월 일별 데이터는 365행 | 월별 요약으로 기본 제공, 일별은 요청 시 |

## 7. 버전 관리

### 7-1. API 변경 시

모바일인덱스 Insight API가 변경되면 MCP 서버도 업데이트해야 합니다.

| 상황 | 영향 | 대응 |
|------|------|------|
| API 파라미터 추가 | MCP 도구에 새 파라미터 추가 필요 | 선택 파라미터로 추가하면 기존 사용에 영향 없음 |
| API 응답 형식 변경 | MCP 응답도 변경됨 | AI가 새 형식을 해석해야 하므로 도구 설명 업데이트 필요 |
| API 엔드포인트 삭제 | 해당 MCP 도구 사용 불가 | 도구 삭제 또는 "더 이상 지원하지 않습니다" 메시지 반환 |
| 새 API 추가 | 새 MCP 도구 추가 가능 | 도구 추가 후 서버 재배포 |

### 7-2. MCP 프로토콜 업데이트

MCP 표준 자체가 업데이트될 수 있습니다 (현재 v1.0).

> 📌 MCP SDK를 최신 버전으로 유지하면 대부분 자동 대응됩니다.
> 📌 큰 변경(breaking change)이 있을 경우 서버 코드 수정 필요.
