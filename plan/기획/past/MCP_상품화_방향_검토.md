# MCP 상품화 방향 검토

> 경쟁사 MCP 제공 현황, 실제 사용 사례, MCP 커스텀 범위, 상품 개발 방향성을 정리한 문서입니다.

<br>

## 1. 경쟁사 MCP 제공 현황

### 1-1. 시밀러웹 (SimilarWeb) — 2025년 9월 출시

시밀러웹은 MCP를 "별도 상품"이 아닌, 기존 API 구독의 확장 채널로 제공하고 있습니다.

| 항목 | 내용 |
|------|------|
| 포지셔닝 | "Make your AI Agents Smarter" — AI 에이전트에 디지털 인텔리전스를 공급하는 데이터 파이프라인 |
| 과금 | 별도 MCP 요금 없음. 기존 API 구독(Business $35,000/년~)에 포함. MCP 호출도 기존 Data Credits 차감 |
| 엔드포인트 | `https://mcp.similarweb.com` (원격 서버) |
| 도구 수 | 23개 (웹사이트 분석 18개 + 검색 분석 2개 + 앱 분석 3개) |
| 연결 지원 | Claude Desktop, Cursor, Microsoft Copilot Studio, Zapier, n8n |
| 인증 | API 키 기반 (기존 Similarweb API 키 그대로 사용) |

시밀러웹이 MCP를 설명하는 핵심 메시지:

> "API가 원시 데이터를 제공한다면, MCP는 AI에게 데이터와 함께 그 데이터를 어떻게 사용해야 하는지 맥락(context)까지 제공한다."
> ([출처: Similarweb MCP 발표 블로그](https://www.similarweb.com/blog/updates/announcements/mcp-server-launch/))

즉, 시밀러웹도 MCP를 "새로운 기능"이 아니라 "기존 데이터를 AI가 더 잘 쓸 수 있게 포장해주는 레이어"로 보고 있습니다.

시밀러웹의 MCP 활용 방향은 크게 3가지입니다:
1. 마케터/리서처가 AI 플랫폼에서 자연어로 경쟁사 분석
2. 개발자가 커스텀 대시보드, 자동 리포팅 시스템에 데이터 연결
3. 노코드 도구(Zapier, n8n)로 자동화 워크플로우 구축


### 1-2. AppsFlyer — 2025년 7월 출시 (베타)

AppsFlyer는 MCP를 "AI 마케팅 오케스트레이션 도구"로 포지셔닝하고 있습니다.

| 항목 | 내용 |
|------|------|
| 포지셔닝 | "Your data team is busy. Your LLM isn't." — 데이터팀 의존도를 줄이고, 마케터가 직접 AI로 데이터에 접근 |
| 과금 | 기존 구독에 포함 (별도 MCP 요금 없음) |
| 도구 범위 | 캠페인 성과 조회, 오디언스 관리, OneLink 감사, 앱 설정 확인, SDK 문서 조회 등 |
| 연결 지원 | Claude, ChatGPT, Cursor, n8n, Zapier 등 |
| 특징 | 오픈소스로 GitHub에 MCP 서버 코드 공개 + 워크플로우 템플릿 제공 |

AppsFlyer가 강조하는 핵심 메시지:

> "No engineering support, no API setup, no waiting for analysts. Ask 'What's driving my CAC increase?' and get answers in seconds."
> ([출처: AppsFlyer MCP 제품 페이지](https://www.appsflyer.com/products/agentic-ai/mcp/))

AppsFlyer는 MCP 자체보다 "MCP + 자동화 도구(n8n 등)를 결합한 워크플로우"에 집중하고 있습니다. GitHub에 바로 쓸 수 있는 워크플로우 템플릿을 제공하며, 사용자가 직접 자동화를 구축하도록 유도합니다.

### 1-3. Braze — 2025년 6월 출시 (베타)

Braze는 MCP를 "대화형 데이터 접근 도구"로 포지셔닝합니다.

| 항목 | 내용 |
|------|------|
| 포지셔닝 | "If you can type a question, you can explore your Braze ecosystem" — 기술 전문성 없이 데이터 탐색 |
| 과금 | 기존 구독에 포함 (별도 MCP 요금 없음) |
| 도구 수 | 38개 (읽기 전용, 15개 카테고리) — 캠페인, 캔버스, 세그먼트, 커스텀 속성 등 |
| 보안 | 읽기 전용(read-only), 비PII 데이터만 반환, 전용 API 키 생성 권장 |
| 연결 지원 | Claude Desktop, Cursor, VS Code 등 |
| 설치 시간 | 10분 이내 |

Braze가 내부 마케팅팀에서 실제로 MCP를 사용한 사례:

> Braze 내부 마케팅팀이 자사 MCP를 수주간 테스트하며 발견한 것들:
> - "FY별 캠페인/캔버스 수를 알려줘" → 연도별 마케팅 활동량 추이 즉시 파악
> - "7월 캠페인 성과를 요약해줘" → 제품 교육 뉴스레터의 오픈율이 가장 높고, 월말 발송이 월중 대비 2배 성과
> - "이 인사이트 기반으로 시작할 것 2개, 계속할 것 2개, 중단할 것 2개를 알려줘" → 데이터 기반 액션 아이템 도출
> - 보너스: 180개 이상의 중복 커스텀 속성과 미사용 세그먼트를 발견하여 정리
>
> ([출처: Braze MCP 소개 블로그](https://www.braze.com/resources/articles/introducing-braze-mcp-server), 내용 재구성)

### 1-4. 경쟁사 공통점 정리

| 공통점 | 설명 |
|--------|------|
| MCP를 별도 상품으로 팔지 않음 | 3곳 모두 기존 구독에 포함. MCP 자체로 추가 매출을 만들지 않음 |
| "기능"이 아닌 "접근 방식"으로 포지셔닝 | "새로운 기능을 만들었다"가 아니라 "기존 데이터에 접근하는 새로운 방법을 열었다"로 설명 |
| 사용자 자율성 강조 | 사용자가 자기 AI 환경에서, 자기 방식대로 데이터를 활용하도록 유도 |
| 워크플로우/자동화와 결합 | 단순 데이터 조회를 넘어, n8n/Zapier 등과 결합한 자동화 시나리오를 적극 제시 |
| 읽기 전용(read-only) 중심 | 데이터 조회에 집중. 데이터 수정/삭제 기능은 제공하지 않음 |

## 2. 실제 사용 사례

### 2-1. AppsFlyer MCP — 주간 UA 리포트 자동화

모바일 게임 UA(User Acquisition) 컨설턴트가 AppsFlyer MCP를 활용해 주간 리포팅을 자동화한 사례입니다.

> "매주 같은 일을 반복했다. AppsFlyer를 열고, 한 클라이언트, 또 한 클라이언트... 10개 계정, 10개 대시보드, 주가 시작되기도 전에 몇 시간이 사라졌다. 이제는 아니다."
> ([출처: gamigion.com](https://www.gamigion.com/stop-doing-ua-reports-i-automated-my-weekly-ua-reporting/), 내용 재구성)

구체적인 자동화 구성:
- AppsFlyer MCP를 n8n(노코드 자동화 도구)에 연결
- 매주 자동으로 10개 클라이언트의 UA 성과 데이터를 수집
- AI가 데이터를 분석하여 리포트 생성
- Slack, 이메일, Notion, Google Sheets 등 원하는 곳으로 자동 전달
- 설정 시간: 15~20분, 코딩 불필요

이 사례의 핵심은 MCP 자체가 아니라, MCP를 자동화 도구와 결합해서 "반복 업무를 없앤 것"입니다.

### 2-2. AppsFlyer MCP — 예산 초과 실시간 모니터링

AppsFlyer가 공식 제공하는 워크플로우 템플릿 중 하나입니다.

- AI 에이전트가 MCP를 통해 매체별 광고 비용을 실시간 모니터링
- 사전 설정한 임계값(예: "Facebook 지출이 $5,000 넘으면") 초과 시 즉시 Slack/이메일 알림
- 예산 초과를 사후가 아닌 사전에 감지

> "Prevent budget disasters: Set up 24/7 monitoring that alerts you before overspend happens, not after."
> ([출처: AppsFlyer 블로그](https://www.appsflyer.com/blog/measurement-analytics/mcp-ai-workflows/))

### 2-3. Braze MCP — 내부 마케팅팀 데이터 탐색

위 1-3에서 언급한 Braze 내부 사례를 좀 더 구체적으로 보면:

- 연도별 캠페인 수 추이 파악 → 리소스 계획 수립
- 월별 캠페인 성과 요약 → 어떤 유형의 캠페인이 효과적인지 패턴 발견
- AI에게 "시작/계속/중단할 것" 추천 요청 → 데이터 기반 의사결정
- 180개 이상의 중복 커스텀 속성 발견 → 데이터 정리(하우스키핑)

Braze가 공유한 MCP 사용 팁:
- 구체적으로 질문할수록 좋은 결과 ("최근 성과" 보다 "2025년 7월 성과"가 낫다)
- AI가 반복하거나 막히면 새 대화를 시작하고 질문을 더 구체적으로
- 분석 결과가 나오면 "가장 임팩트 큰 것", "가장 빠른 성과", "장기적으로 시간을 아끼는 것" 등으로 우선순위를 요청
- CMO 보고용 슬라이드 요약, Slack 공유용 시각화 등도 요청 가능

### 2-4. 시밀러웹 MCP — ShopVision AI 에이전트 연동

ShopVision(이커머스 AI 플랫폼)이 시밀러웹 API/MCP를 활용한 사례입니다.

> "시밀러웹은 경쟁사 행동의 영향에 대한 세밀한 디지털 인사이트를 제공하여 기존 데이터셋을 강화한다. 이제 경쟁사 트래픽 급증과 고객사 매출 성과를 실시간으로 상관분석할 수 있다."
> — Peter Sheldon, CPO & Co-Founder, ShopVision
> ([출처: Similarweb 고객 사례](https://www.similarweb.com/corp/clients/shopvision-success-story/), 내용 재구성)

활용 방식:
- 시밀러웹의 트래픽/참여 데이터를 ShopVision AI 플랫폼에 직접 연동
- 경쟁사 프로모션 → 실제 트래픽 변화 상관관계 분석
- 고객사의 Shopify/GA4 자사 데이터와 시밀러웹 경쟁사 데이터를 결합
- 수일 걸리던 경쟁사 리서치를 수초 만에 AI가 처리

이 사례의 핵심: MCP/API를 통해 시밀러웹 데이터가 "시밀러웹 대시보드"가 아닌 "고객의 자체 AI 플랫폼" 안에서 활용되고 있다는 점입니다.

### 2-5. 사례에서 보이는 패턴

| 패턴 | 설명 |
|------|------|
| 반복 업무 자동화 | 주간 리포트, 일일 모니터링 등 반복적인 데이터 수집+분석 업무를 AI+MCP로 자동화 |
| 대시보드 → 대화형 전환 | 대시보드에 들어가서 필터 걸고 데이터 뽑는 대신, AI에게 자연어로 질문 |
| 자사 데이터 + 외부 데이터 결합 | MCP로 가져온 외부 데이터를 자사 데이터와 합쳐서 더 깊은 분석 |
| 비기술 직군의 데이터 접근 | SQL이나 API를 모르는 마케터/기획자도 데이터에 직접 접근 가능 |

## 3. MCP 개발 시 커스텀 가능 범위

MCP는 "포장지"라는 표현이 맞습니다. 정확히는 **우리 데이터를 AI가 잘 쓸 수 있도록 포장하는 레이어**입니다.
하지만 이 포장지를 어떻게 만드느냐에 따라 사용자 경험이 크게 달라집니다.

MCP 서버 개발 시 우리가 제어할 수 있는 것과 없는 것을 명확히 구분합니다.

### 3-1. 우리가 제어할 수 있는 것

MCP 프로토콜은 서버 개발자에게 3가지 핵심 구성 요소를 제공합니다: **도구(Tools)**, **리소스(Resources)**, **프롬프트(Prompts)**.
([참고: WorkOS MCP 가이드](https://workos.com/blog/mcp-features-guide))

**① 도구(Tools) — AI가 실행할 수 있는 기능**

| 커스텀 항목 | 설명 | 예시 |
|------------|------|------|
| 도구 이름 | AI가 도구를 식별하는 이름 | `app_usage`, `competitor_analysis` |
| 도구 설명 | AI가 "이 도구를 언제 써야 하는지" 판단하는 근거. 이 설명의 품질이 사용자 경험을 좌우 | "특정 앱의 월간 활성 사용자 수(MAU), 일간 활성 사용자 수(DAU), 사용 시간 추이를 조회합니다" |
| 파라미터 설계 | 어떤 입력값을 받을지, 필수/선택 구분, 기본값, 형식 | `app_name`(필수), `period`(선택, 기본값: 최근 3개월) |
| 응답 포맷 | AI에게 돌려주는 데이터의 구조와 내용 | JSON 원본, 요약 텍스트 + 상세 데이터, 출처/기준일 포함 여부 |
| 응답 가이드 | 응답에 포함하는 안내 문구로 AI의 해석 방향을 유도 | "이 데이터는 모바일인덱스 기준이며, 데이터에 없는 내용은 추측하지 마세요" |
| 에러 메시지 | 데이터가 없거나 오류 시 AI에게 전달하는 메시지 | "해당 앱의 데이터가 없습니다. 앱 이름을 다시 확인해주세요" |
| 도구 조합 설계 | 단일 API 1:1 매핑 vs 여러 API를 묶은 복합 도구 | "경쟁앱 종합 분석" 도구 = 사용량 + 이탈 + 페르소나를 한 번에 호출 |

> 📌 핵심: **도구 설명(description)이 MCP의 품질을 결정합니다.**
> AI는 도구 설명을 읽고 "이 질문에 어떤 도구를 쓸지"를 판단합니다.
> 설명이 모호하면 AI가 엉뚱한 도구를 선택하거나, 적절한 도구를 못 찾습니다.
> 이것이 단순 API와 MCP의 가장 큰 차이점이며, 우리가 가장 공을 들여야 하는 부분입니다.

**② 리소스(Resources) — AI가 참조할 수 있는 정보**

도구와 달리 "실행"하는 것이 아니라, AI가 "읽을 수 있는" 정적 데이터입니다.

| 활용 예시 | 설명 |
|----------|------|
| 업종 분류 체계 | 모바일인덱스의 업종 대분류/소분류 목록을 리소스로 제공 → AI가 사용자 질문에서 업종을 자동 매핑 |
| 데이터 사전 | 각 지표의 정의(MAU란 무엇인가, DAU 산출 기준 등)를 리소스로 제공 → AI가 정확한 맥락으로 답변 |
| 조회 가능 기간 | "데이터는 2023년 1월부터 현재까지 조회 가능" 같은 메타 정보 |

**③ 프롬프트(Prompts) — 미리 만들어둔 질의 템플릿**

사용자가 자주 하는 질문 패턴을 템플릿으로 제공할 수 있습니다.

| 활용 예시 | 설명 |
|----------|------|
| "경쟁앱 분석" 프롬프트 | 앱 이름만 입력하면, 사용량 + 이탈 + 페르소나 + 경쟁사 비교를 순차적으로 실행하는 템플릿 |
| "업종 리포트" 프롬프트 | 업종명만 입력하면, 시장 규모 → Top 앱 → 추이 분석 → 인사이트 도출까지 자동 실행 |
| "월간 보고서" 프롬프트 | 앱 이름 + 기간만 입력하면, 정해진 포맷의 월간 보고서를 생성 |

> 📌 프롬프트는 "사용자가 MCP를 어떻게 써야 하는지 모를 때" 가이드 역할을 합니다.
> 시밀러웹도 공식 문서에 "경쟁 분석 대시보드 만들어줘", "트래픽 소스 분석해줘" 같은 예시 프롬프트를 제공합니다.

### 3-2. 우리가 제어할 수 없는 것

| 항목 | 설명 |
|------|------|
| AI의 답변 스타일 | 같은 데이터를 줘도 Claude는 이렇게, ChatGPT는 저렇게 표현. 이건 AI 플랫폼이 결정 |
| AI의 추론/분석 능력 | 데이터를 받은 후 어떤 인사이트를 도출하는지는 AI 모델의 역량에 의존 |
| 할루시네이션 완전 차단 | 응답에 가이드를 넣어 최소화할 수 있지만, 100% 차단은 불가능 |
| AI 플랫폼의 UI/UX | 사용자가 MCP를 설정하는 과정, 대화 인터페이스 등은 각 AI 플랫폼이 결정 |
| AI 플랫폼의 장애/정책 변경 | AI 플랫폼 자체의 장애, MCP 지원 정책 변경 등은 대응 불가 |

### 3-3. "포장지"를 잘 만든다는 것의 의미

MCP가 포장지라면, 포장지의 품질은 다음으로 결정됩니다:

```
① 도구 설명의 품질 → AI가 올바른 도구를 선택하는가
② 응답 포맷의 설계 → AI가 데이터를 정확히 해석하는가
③ 에러 처리의 친절함 → 문제 상황에서 사용자가 막히지 않는가
④ 프롬프트 템플릿 → 사용자가 "뭘 물어봐야 하는지" 알 수 있는가
⑤ 리소스(메타 정보) → AI가 우리 데이터의 맥락을 이해하는가
```

같은 API 데이터라도 MCP를 어떻게 설계하느냐에 따라:
- "카카오톡 MAU 알려줘" → 숫자만 던지는 MCP vs 추이 + 전월 대비 + 업종 내 순위까지 함께 주는 MCP
- "증권 업종 분석해줘" → 도구를 못 찾아 헤매는 MCP vs 업종 코드 자동 매핑 → 순위 → 추이 → 인사이트까지 흐르는 MCP

**결론: MCP는 포장지가 맞지만, 포장지의 설계가 곧 상품의 품질입니다.**

<br>


## 4. 왜 써야 하는가 — 사용자 관점의 문제 정의

### 4-1. 현재 사용자의 워크플로우

모바일인덱스 고객이 "업종 분석 보고서"를 만든다고 가정합니다.

```
현재 워크플로우 (도구 3~4개를 왔다갔다)

① 시장 조사
   └─ 웹 서치 (구글, 뉴스, 업계 리포트)
   └─ 시장 트렌드, 업계 동향 파악

② 데이터 수집
   └─ 모바일인덱스 대시보드 접속
   └─ 업종 선택 → 앱 순위 확인 → 필터 설정 → 데이터 다운로드
   └─ 경쟁앱 하나씩 들어가서 MAU, 사용시간, 데모그래픽 확인
   └─ 스크린샷 찍거나 CSV 다운로드

③ 데이터 정리
   └─ 엑셀에 데이터 옮기기
   └─ 차트 만들기
   └─ 앱별 비교표 정리

④ 보고서 작성
   └─ 워드/PPT 열기
   └─ 차트 붙여넣기
   └─ 인사이트 작성
   └─ 전략 제안 작성
```

이 과정에서 사용자가 겪는 문제:

| 문제 | 설명 |
|------|------|
| 시간 | 단순 데이터 수집 + 정리에 전체 작업 시간의 60~70%를 소모 |
| 도구 전환 비용 | 웹 → 모바일인덱스 → 엑셀 → PPT를 왔다갔다하며 컨텍스트가 끊김 |
| 수동 반복 | 경쟁앱 5개를 분석하려면 같은 조작을 5번 반복 |
| 데이터 복사 오류 | 수동으로 옮기는 과정에서 숫자 오류, 기간 불일치 등 발생 가능 |
| 분석의 깊이 한계 | 데이터 수집에 지쳐서 정작 인사이트 도출에 시간을 못 씀 |

### 4-2. MCP가 바꾸는 워크플로우

```
MCP 워크플로우 (AI 플랫폼 하나에서 끝)

사용자: "증권 업종 앱 시장 분석 보고서 만들어줘"

AI가 하나의 대화 안에서 연속 실행:
  ① 업종 코드 자동 매핑 (MCP 도구: 업종 분류 조회)
  ② 업종 전체 MAU 추이 조회 (MCP 도구: 시장 규모)
  ③ Top 앱 순위 + 점유율 (MCP 도구: 사용량 순위)
  ④ 주요 앱 상세 비교 (MCP 도구: 앱별 사용량, 데모그래픽)
  ⑤ 이탈/유입 분석 (MCP 도구: 이탈 분석)
  ⑥ 데이터 종합 → 인사이트 도출 → 보고서 작성 (AI의 분석 능력)

결과: 데이터 수집 + 정리 + 차트 + 인사이트 + 보고서가 하나의 흐름으로 완성
```

핵심은 **분절되어 있던 작업이 하나의 연속된 흐름이 된다**는 것입니다.

| 기존 | MCP 이후 |
|------|---------|
| 웹 서치 → 모바일인덱스 → 엑셀 → PPT (4개 도구) | AI 플랫폼 하나 |
| 데이터 수집에 60~70% 시간 소모 | 데이터 수집은 AI가 자동, 사용자는 인사이트에 집중 |
| 경쟁앱 5개 = 같은 조작 5번 반복 | "경쟁앱 5개 비교해줘" 한 마디 |
| 데이터 복사 과정에서 오류 가능 | AI가 직접 데이터를 받아서 처리, 복사 오류 없음 |
| 분석 깊이가 시간에 제한됨 | 데이터 수집 시간이 0에 가까워져 분석에 집중 가능 |

### 4-3. 구체적 사용 시나리오

**시나리오 1: 업종 분석을 막 시작할 때**

```
사용자: "핀테크 업종 앱 시장이 어떻게 돌아가는지 전체적으로 파악하고 싶어"

AI + MCP가 하는 일:
→ 핀테크 관련 업종 분류 조회
→ 업종 전체 MAU/설치 추이 (시장 성장세 파악)
→ Top 20 앱 순위 + 점유율
→ 주요 앱 3~5개의 사용 패턴 비교
→ "이 업종은 최근 6개월간 MAU가 12% 성장했고, 토스가 점유율 1위를 유지하는 가운데
    카카오뱅크가 빠르게 추격 중입니다. 주요 성장 동력은..." 형태의 요약
```

기존이라면: 모바일인덱스에서 업종 필터 → 순위 확인 → 앱 하나씩 클릭 → 데이터 메모 → 정리. 최소 30분~1시간.

**시나리오 2: 내 앱과 경쟁 앱의 보고 싶은 데이터가 명확할 때**

```
사용자: "카카오톡과 라인의 최근 6개월 MAU, DAU, 사용시간을 비교해줘"

AI + MCP가 하는 일:
→ 카카오톡 사용량 조회 (6개월)
→ 라인 사용량 조회 (6개월)
→ 두 앱의 MAU/DAU/사용시간을 월별로 나란히 비교
→ 차이점과 추세 요약
```

기존이라면: 모바일인덱스에서 카카오톡 검색 → 기간 설정 → 데이터 확인 → 라인 검색 → 같은 작업 반복 → 엑셀에 옮겨서 비교표 작성. 최소 15~20분.

**시나리오 3: 보고용 차트와 데이터 나열이 필요할 때**

```
사용자: "다음 주 경영진 보고에 쓸 증권앱 시장 현황 차트를 만들어줘.
        Top 5 앱 MAU 추이, 연령대별 사용자 분포, 이탈률 비교가 필요해"

AI + MCP가 하는 일:
→ 증권 업종 Top 5 앱 MAU 12개월 추이 데이터 수집
→ 각 앱의 연령대별 사용자 분포 조회
→ 이탈률 데이터 조회
→ 차트 3개 생성 + 핵심 수치 요약 + 보고서 형태로 정리
```

기존이라면: 모바일인덱스에서 5개 앱 각각 데이터 수집 → 엑셀에서 차트 3개 제작 → PPT에 옮기기 → 코멘트 작성. 최소 1~2시간.

### 4-4. "왜 써야 하는가"에 대한 답

MCP는 새로운 기능이 아닙니다. 모바일인덱스 데이터는 이미 있고, AI 플랫폼도 이미 있습니다.

MCP가 하는 것은 **이 둘을 연결해서, 사용자의 워크플로우에서 "데이터 수집 + 정리"라는 가장 시간이 많이 드는 단계를 없애는 것**입니다.

```
Before: 시장 조사(웹) → 데이터 수집(모바일인덱스) → 정리(엑셀) → 보고서(PPT)
                         ↑ 여기가 전체 시간의 60~70% ↑

After:  AI에게 질문 → 데이터 수집+정리+분석+보고서가 하나의 흐름으로 완성
                      ↑ 이 과정이 자동 ↑
```

사용자가 MCP를 써야 하는 이유는 단순합니다:
**"모바일인덱스 대시보드에 들어가서 클릭하고 다운로드하는 시간을 없애고, 그 시간에 분석과 의사결정에 집중할 수 있으니까."**

<br>


## 5. MCP 상품 개발 방향성

### 5-1. MCP는 "기능"이 아니라 "접근 방식"이다

경쟁사 3곳(시밀러웹, AppsFlyer, Braze) 모두 MCP를 별도 상품으로 팔지 않습니다.
MCP를 "새로운 기능"으로 포장하는 것은 MCP의 본질과 맞지 않습니다.

MCP의 본질:
```
우리 데이터를 사용자가 이미 쓰고 있는 AI 플랫폼에 연결해주는 파이프라인.
사용자가 자기 환경에서, 자기 방식대로, 우리 데이터를 활용할 수 있게 해주는 것.
```

따라서 상품 개발 방향은 "MCP라는 기능을 만든다"가 아니라:
**"모바일인덱스 데이터를 AI 시대에 맞게 제공하는 방식을 추가한다"**가 되어야 합니다.

### 5-2. 상품 포지셔닝 제안

```
기존: 모바일인덱스 = 대시보드에서 데이터를 보는 서비스
추가: 모바일인덱스 = 대시보드 + AI에서 데이터를 쓰는 서비스
```

MCP는 모바일인덱스의 "데이터 전달 채널"이 하나 추가되는 것입니다:

| 채널 | 사용자 | 사용 방식 |
|------|--------|----------|
| 웹 대시보드 | 직접 조작하는 사용자 | 로그인 → 필터 → 차트 확인 → 다운로드 |
| API | 개발자 | 코드로 데이터 호출 → 자체 시스템에 연동 |
| MCP (신규) | AI 플랫폼 사용자 | AI에게 자연어로 질문 → 데이터 자동 수집 + 분석 |

### 5-3. 과금 방향

경쟁사 사례와 MCP의 본질을 고려하면:

| 방향 | 설명 | 근거 |
|------|------|------|
| 기존 구독에 포함 (권장) | 프리미엄 이상 고객에게 MCP 접근 포함 | 시밀러웹/AppsFlyer/Braze 모두 이 방식. MCP 자체로 매출을 만들기보다, 기존 구독의 가치를 높여 이탈 방지 + 업셀링 유도 |
| 프리미엄 전용 기능 | 에센셜 고객은 MCP 미제공, 프리미엄 이상만 제공 | 에센셜 → 프리미엄 업그레이드 유도 장치로 활용 |
| MCP 별도 상품 (비권장) | MCP만 따로 판매 | 경쟁사 대비 차별화 어려움. 사용자 입장에서 "왜 따로 돈을 내야 하지?" |

### 5-4. 개발 우선순위

MCP 상품의 품질은 "도구를 몇 개 만드느냐"가 아니라 "도구의 설계 품질"로 결정됩니다.

**Phase 1: 도구 품질 (베타)**
- 27개 도구의 설명(description) 품질 고도화 — AI가 올바른 도구를 선택하도록
- 응답 포맷 최적화 — 핵심 수치 요약 + 상세 데이터 병행
- 에러 메시지 친절화 — 사용자가 막히지 않도록
- 할루시네이션 방지 가이드 응답에 포함

**Phase 2: 사용자 경험 (정식 출시)**
- 프롬프트 템플릿 제공 — "경쟁앱 분석", "업종 리포트", "월간 보고서" 등
- 리소스 제공 — 업종 분류 체계, 데이터 사전, 조회 가능 기간 등
- 복합 도구 추가 — 사용 패턴 분석 후, 자주 조합되는 도구를 하나로 묶기
- 연동 가이드 고도화 — AI 플랫폼별 + 사용 시나리오별 가이드

**Phase 3: 생태계 확장 (중장기)**
- n8n/Zapier 등 자동화 도구 연동 가이드 및 템플릿 제공
- 고객사별 커스텀 프롬프트/워크플로우 컨설팅
- 사용 패턴 데이터 기반으로 새로운 복합 도구 개발

### 5-5. 핵심 메시지 제안

내부 보고 및 고객 커뮤니케이션에 사용할 수 있는 핵심 메시지:

**내부용 (경영진/유관부서):**
> "MCP는 새로운 기능이 아니라, 모바일인덱스 데이터의 새로운 전달 채널입니다.
> 사용자가 대시보드 대신 AI 플랫폼에서 우리 데이터를 쓸 수 있게 해주는 것이며,
> 이를 통해 기존 구독의 가치를 높이고, 프리미엄 업그레이드를 유도합니다."

**고객용:**
> "이제 모바일인덱스 데이터를 AI에서 바로 쓸 수 있습니다.
> 대시보드에 들어가서 클릭하고 다운로드하는 대신,
> AI에게 물어보면 데이터 수집부터 분석, 보고서까지 하나의 흐름으로 완성됩니다."


## 6. 전사 확장 — IGAW 풀 퍼널 MCP 전략

### 6-1. 왜 전사 확장인가

MCP의 기술적 특성상, AI 플랫폼 하나에 여러 MCP 서버를 동시에 연결할 수 있습니다.
AI가 사용자의 질문 맥락을 보고, 어떤 MCP 서버의 어떤 도구를 쓸지 자동으로 판단합니다.

IGAW는 3개 서비스가 각각 데이터 퍼널의 다른 단계를 담당합니다:

```
시장은 어떤가        →  광고를 어떻게 집행하나  →  고객이 어떻게 반응하나
(모바일인덱스)          (트레이딩웍스)              (디파이너리)
 시장 데이터              광고 집행 데이터            고객 행동 데이터
```

이 3개를 각각 MCP로 만들면, 사용자가 AI 하나에서 풀 퍼널 데이터를 끊김 없이 활용할 수 있습니다.
이런 조합이 가능한 회사는 시장에서 IGAW뿐입니다.

### 6-2. 전사 MCP 확장 시 사용자 경험

**단일 서비스 MCP (현재 모바일인덱스만)**
```
사용자: "증권 업종 앱 시장 분석해줘"
AI: 모바일인덱스 MCP → 시장 데이터 조회 → 분석 결과 제공
    (여기서 끝. 광고 성과나 고객 행동은 별도 서비스에서 따로 확인해야 함)
```

**전사 MCP (모바일인덱스 + 트레이딩웍스 + 디파이너리)**
```
사용자: "우리 앱의 시장 포지션, 광고 효율, 고객 리텐션을 종합 분석해줘"

AI가 하나의 대화에서 3개 MCP를 자동 조합:
  ① 모바일인덱스 MCP → 업종 내 순위, 경쟁앱 대비 MAU/점유율, 시장 트렌드
  ② 트레이딩웍스 MCP → 매체별 광고 성과(ROAS, CTR), 예산 효율, 소재별 성과
  ③ 디파이너리 MCP  → 유입 경로별 리텐션, 이탈 위험 세그먼트, LTV 분석

  → 3개 데이터를 종합하여:
     "시장에서 3위인데, 광고 효율은 1위 대비 높음. 그러나 D7 리텐션이 낮아
      광고로 유입된 사용자가 이탈하는 구조. 리텐션 개선이 최우선 과제"
```

핵심: 사용자가 3개 서비스 대시보드를 따로 열 필요 없이, AI 하나에서 풀 퍼널 인사이트를 얻습니다.

### 6-3. 서비스별 MCP 역할

| 서비스 | MCP 역할 | 데이터 영역 | 주요 질의 예시 |
|--------|---------|-----------|--------------|
| 모바일인덱스 | 시장 인텔리전스 | 외부 시장 데이터 (경쟁사 포함) | "증권 업종 Top 10 앱 MAU 추이", "경쟁앱 사용자 이탈 분석" |
| 트레이딩웍스 | 광고 성과 분석 | 광고 집행 데이터 | "매체별 ROAS 비교", "CTR 낮은 소재 찾아줘", "예산 재배분 안" |
| 디파이너리 | 고객 행동 분석 | 자사 고객 데이터 | "리텐션 캠페인 성과", "이탈 위험 세그먼트", "푸시 A/B 테스트 결과" |

### 6-4. 전사 확장 시나리오

**시나리오 1: 마케팅 팀장의 주간 보고**

```
사용자: "이번 주 마케팅 성과를 종합 정리해줘"

AI 자동 실행:
  모바일인덱스 MCP → 우리 앱의 업종 내 순위 변동, 경쟁앱 동향
  트레이딩웍스 MCP → 이번 주 매체별 광고 지출/성과, 예산 소진율
  디파이너리 MCP  → 신규 유입 사용자 리텐션, 캠페인별 전환율

  → 종합 주간 보고서 자동 생성
```

**시나리오 2: 신규 앱 런칭 전략**

```
사용자: "핀테크 앱을 새로 런칭하려는데, 시장 진입 전략을 세워줘"

AI 자동 실행:
  모바일인덱스 MCP → 핀테크 시장 규모, 경쟁 구도, 사용자 데모그래픽, 이탈 패턴
  트레이딩웍스 MCP → 핀테크 업종 광고 단가 벤치마크, 효율 높은 매체/소재 유형
  디파이너리 MCP  → 유사 앱의 온보딩 퍼널 전환율, 초기 리텐션 벤치마크

  → 시장 분석 + 광고 전략 + 리텐션 전략을 포함한 런칭 전략 보고서
```

**시나리오 3: 경영진 대시보드**

```
사용자: "우리 앱의 이번 달 전체 퍼포먼스를 한눈에 보여줘"

AI 자동 실행:
  모바일인덱스 MCP → 시장 점유율 변화, 경쟁사 대비 포지션
  트레이딩웍스 MCP → 광고 ROI, 예산 효율, 매체 포트폴리오
  디파이너리 MCP  → MAU/DAU 추이, 리텐션, LTV, 핵심 이벤트 전환율

  → 시장 → 획득 → 유지의 풀 퍼널 월간 리포트
```

### 6-5. 전사 확장 로드맵

```
Phase 1: 모바일인덱스 MCP (현재)
├── 시장 데이터 27개 도구
├── 베타 테스트 → 정식 출시
└── MCP 서버 설계 패턴 확립 (도구 설계, 응답 포맷, 인증 등)
    ↓
    이 단계에서 확립한 설계 패턴을 다음 서비스에 그대로 적용

Phase 2: 트레이딩웍스 MCP
├── 광고 집행 데이터 도구 (캠페인 성과, 매체별 분석, 예산 모니터링 등)
├── 모바일인덱스 MCP와 크로스 서비스 시나리오 검증
└── "시장 데이터 + 광고 데이터" 결합 분석 가능

Phase 3: 디파이너리 MCP
├── 고객 행동 데이터 도구 (리텐션, 세그먼트, 캠페인 성과, LTV 등)
├── 3개 MCP 풀 퍼널 연동 검증
└── "시장 → 광고 → 고객" 풀 퍼널 분석 완성
```

> 📌 모바일인덱스에서 MCP 서버 설계 패턴(도구 네이밍, 응답 포맷, 인증 구조, 에러 처리 등)을 잘 잡아두면,
> 트레이딩웍스와 디파이너리는 같은 패턴을 적용하면 되므로 개발 속도가 빨라집니다.
> 모바일인덱스 MCP가 전사 MCP 전략의 "파일럿"이자 "설계 표준"이 되는 셈입니다.

### 6-6. 전사 확장의 경쟁 우위

| 항목 | IGAW (전사 MCP) | 경쟁사 |
|------|----------------|--------|
| 데이터 범위 | 시장 + 광고 + 고객 (풀 퍼널) | 각자 영역만 (시밀러웹=시장, AppsFlyer=어트리뷰션, Braze=CRM) |
| 크로스 분석 | AI가 3개 MCP를 자동 조합하여 풀 퍼널 인사이트 도출 | 서로 다른 회사 MCP를 조합해야 하며, 데이터 기준/정의가 달라 정합성 문제 |
| 사용자 경험 | 하나의 IGAW 계정으로 3개 MCP 모두 접근 | 서비스마다 별도 계정/API 키/과금 |
| 데이터 정합성 | 동일 회사 데이터이므로 기준 통일 가능 | 회사마다 MAU 정의, 기간 기준 등이 다름 |

**핵심 메시지 (전사 확장용):**
> "IGAW MCP는 시장 분석(모바일인덱스) → 광고 최적화(트레이딩웍스) → 고객 분석(디파이너리)까지,
> AI 하나에서 풀 퍼널 데이터를 끊김 없이 활용할 수 있는 유일한 솔루션입니다."
