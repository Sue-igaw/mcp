**메인:**
모바일인덱스 MCP
**서브:**
데이터를 보여주는 것을 넘어, 데이터가 만든 결과물을 제공하는 새로운 시작

## 장표 2
**목차:**
01 문제 인식
**메인:**
데이터는 많지만, 제대로 활용되지 않는다
**서브:**
모바일인덱스는 국내 대표 모바일 앱 데이터 플랫폼이지만, 많은 고객이 데이터를 조회하는 수준에 머무르고 있습니다.
**콘텐츠:**
| 문제 | 설명 |
|------|------|
| 단순 조회 중심 | 월 300만 원을 내고 MAU 숫자 확인 수준에 머무름 |
| 사용 편중 | 연간 계약이지만 보고서 시즌에만 집중 사용 |
| ROI 불명확 | 광고비는 ROAS로 증명 가능하지만, 데이터 구독은 성과 연결이 어려움 |
**결론:**
이것은 고객의 문제가 아니라 **데이터 활용 구조의 한계**

## 장표 3
**목차:**
01 문제 인식
**메인:**
고객이 원하는 것은 데이터가 아니라 결과물
**서브:**
고객은 데이터를 해석하는 과정 자체를 부담스러워합니다. 이미 정리된 인사이트와 실행 가능한 결과물을 더 선호합니다.
**콘텐츠:**
| 현재 | 전환 |
|------|------|
| 데이터 제공 → 고객이 해석 → ROI 불명확 → 예산 삭감 | 결과물 제공 → AI가 해석 → ROI 측정 가능 → 구독 유지 |
**결론:**
데이터 제공을 넘어 **결과물 제공 구조로 전환해야 하며, 그 핵심 수단은 LLM** 입니다.

## 장표 4
**목차:**
01 문제 인식
**메인:**
데이터를 활용하는 과정이 지나치게 복잡하다
**서브:**
사용자는 데이터를 확인하는 것보다, 정리하고 가공하는 데 더 많은 시간을 쓰고 있습니다.
**콘텐츠:**
현재 워크플로우:
```
① 시장 조사      → 웹 서치
② 데이터 수집    → 모바일인덱스 대시보드에서 앱 하나씩 클릭, 필터, 다운로드
③ 데이터 정리    → 엑셀에서 차트 만들고 비교표 정리
④ 보고서 작성    → PPT/워드에서 인사이트 작성
```

| 문제 | 설명 |
|------|------|
| 시간 소모 | 수집 + 정리에 전체 작업 시간의 60~70% |
| 도구 분산 | 웹 → 모바일인덱스 → 엑셀 → PPT를 왔다갔다 |
| 반복 작업 | 경쟁앱 5개 분석 = 같은 조작 5번 반복 |
| 인사이트 부족 | 수집에 지쳐서 분석 단계까지 도달하기 어려움 |
**결론:**
사용자는 데이터를 보기 전에 이미 작업에 지치고 있습니다.

## 장표 5
**목차:**
02 시장 변화
**메인:**
데이터 분석 방식이 변화하고 있다
**서브:**
사용자는 더 이상 대시보드에 접속해서 데이터를 찾지 않습니다. AI에게 질문하고 결과를 바로 얻는 방식으로 전환되고 있습니다.
**콘텐츠:**
| 지표 | 수치 |
|------|------|
| 한국 직장인 AI 사용 경험 | 63.5% |
| 업무 목적 AI 사용률 | 51.8% |
| 한국 업무용 AI 도입률 | 미국(26.5%)의 약 2배 |

- 경쟁사 시밀러웹은 2025.09 MCP 정식 출시
- AppsFlyer·Braze도 베타 운영 중
- **모바일 앱 심화 데이터를 MCP로 제공하는 서비스는 아직 시장에 없음**

출처: 한국은행 "AI의 빠른 확산과 생산성 효과" 보고서, 2025.08
**결론:**
데이터의 문제가 아니라 **데이터를 사용하는 방식이 바뀌고** 있습니다.

## 장표 7
**목차:**
03 해결 방법
**메인:**
이 전환을 구현하는 방법은 두 가지
**서브:**
하나를 선택하는 문제가 아니라, 각 방법의 역할과 실행 순서를 정의하는 것이 중요합니다.
**콘텐츠:**
| | 방법 A — MCP 상품 출시 | 방법 B — 자체 AI 기능 탑재 |
|---|---|---|
| 개념 | 사용자가 이미 쓰는 AI 플랫폼에 모바일인덱스 데이터를 연결 | 모바일인덱스 안에 AI 채팅/분석 기능을 직접 구축 |
| 목적 | 확산 — AI 플랫폼 전체 사용자에게 도달 | 통제 — 완전히 제어된 UX 제공 |
| 특징 | 빠른 실행, 낮은 비용, 넓은 도달 | 완전한 UI/UX 제어, 가이드 UI 구현 가능 |
**결론:**
MCP는 **확산**을 위한 전략이고, 자체 AI는 **통제**를 위한 전략입니다.

## 장표 8
**목차:**
03 해결 방법
**메인:**
방법 A vs 방법 B 비교
**콘텐츠:**
| 항목 | A. MCP 상품 출시 | B. 자체 AI 기능 탑재 |
|------|-----------------|-------------------|
| 개발 범위 | MCP 서버 (중간 레이어) | AI 모델 연동 + 채팅 UI + 프롬프트 설계 |
| 개발 기간 | **6~8주** | 12~16주 이상 |
| 도달 범위 | **Claude/ChatGPT/Gemini 전체 사용자** | 모바일인덱스 기존 고객만 |
| AI 품질 | Claude, GPT 등 최고 수준 AI 활용 | 자체 구현 수준에 의존 |
| UI/UX 제어 | AI 플랫폼에 의존 (제한적) | **완전 제어 가능** |
| 비용 | **서버 운영비만** | AI API 호출 비용 + 개발비 |
| 검증 속도 | **빠르게 시장 반응 확인 가능** | 출시까지 시간이 걸려 검증이 늦음 |

## 장표 9
**목차:**
03 해결 방법
**메인:**
MCP, 이미 만들어진 AI Agent를 활용
**서브:**
자체 AI를 만들려면 LLM 연동부터 채팅 UI까지 전부 직접 개발해야 합니다. MCP를 만들면 이 모든 것을 Claude, ChatGPT, Gemini가 대신 해줍니다.
**콘텐츠:**
| | 자체 AI | MCP |
|---|---|---|
| LLM 연동 | 직접 API 호출 구현 | **AI 플랫폼이 이미 내장** |
| 질문 해석 → 도구 선택 | 오케스트레이션 로직 직접 개발 | **AI가 도구 설명을 읽고 자동 판단** |
| 멀티턴 대화 | 세션/컨텍스트 관리 직접 구현 | **AI 플랫폼이 이미 지원** |
| 채팅 UI | 프론트엔드 직접 개발 | **Claude/ChatGPT UI를 그대로 사용** |
| 우리가 하는 일 | 위 전부 + 데이터 연동 | **도구 27개 정의 + 설명 작성** |
**결론:**
Anthropic, OpenAI, Google이 수천억 원 투자해서 만든 AI 인프라를 그대로 가져다 쓰는 것

## 장표 10
**목차:**
03 해결 방법
**메인:**
모바일인덱스, 쉽게 MCP 상품으로 출시 가능
**콘텐츠:**
1. **즉시 실행 가능** — MCP 서버 27개 도구 개발 완료. 서버 인프라만 세팅하면 베타 출시 가능
2. **사용자 행동 데이터 확보** — 베타에서 사용 패턴을 수집하면 자체 AI 설계에 직접 활용 가능
3. **시장 타이밍 확보** — 시밀러웹이 이미 MCP를 출시한 상황에서 선점 기회를 놓치면 후발 주자
4. **도달 범위 확대** — Claude/ChatGPT/Gemini 전체 사용자가 잠재 고객
5. **리스크 최소화** — 도구 설계, 응답 포맷, 프롬프트 템플릿은 자체 AI에 그대로 재활용 가능
**결론:**
MCP는 단순한 기능이 아니라 **자체 AI를 위한 검증 단계**

## 장표 13
**목차:**
04 무엇이 바뀌는가
**메인:**
어떤 데이터를 봐야 할지 고민할 필요가 없다, 질문만 한다
**서브:**
MCP를 연결하면 데이터 분석의 시작부터 AI와 함께합니다. 사용자는 현재 상황만 말하면, AI가 어떤 데이터를 봐야 하는지 판단하고 직접 가져옵니다.
**콘텐츠:**
| 현재 상황별 질의 예시 | AI가 하는 일 |
|----------|------------|
| "우리 앱 MAU가 떨어지고 있어" | 경쟁앱 MAU 추이 비교 → 이탈 분석 → 원인 진단 |
| "신규 앱 출시를 준비 중이야" | 타겟 업종 시장 규모 → 경쟁 구도 → 포지셔닝 제안 |
| "마케팅 예산을 어디에 써야 할지 모르겠어" | 데모그래픽 분석 → 경쟁앱 사용자 비교 → 타겟 세그먼트 추천 |
| "경영진 보고가 내일이야" | 시장 현황 → 자사 포지션 → 경쟁사 동향 → 보고서 자동 생성 |
| "카카오톡 MAU 알려줘" | MAU 데이터를 자동 조회하여 응답 |
| "증권 업종 앱 순위 TOP 20" | 업종 분류 자동 매핑 → 사용량 순위 조회 → 순위표 생성 |
**결론:**
AI를 기존처럼 쓰면서, **데이터 기반 의사결정이 자동으로 따라온다**

## 장표 14
**목차:**
04 무엇이 바뀌는가
**메인:**
"복붙 AI"와 "연결된 AI"는 다르다
**서브:**
ChatGPT에 데이터를 복붙해도 분석은 가능합니다. 하지만 그 과정이 전부 사용자의 몫입니다. MCP는 그 과정을 없앱니다.
**콘텐츠:**
| | 상용 AI에 직접 복붙 | MCP 연결 |
|---|---|---|
| 데이터 수집 | 사용자가 직접 찾아서 복사 | **AI가 자동으로 가져감** |
| 데이터 범위 | 복사한 만큼만 | **27개 도구 전체 데이터에 접근** |
| 최신성 | 복사 시점 기준 | **항상 실시간 최신 데이터** |
| 반복 작업 | 매번 복사 → 붙여넣기 반복 | **"지난달이랑 비교해줘" 한 마디** |
| 복합 분석 | 사용자가 직접 조합 | **AI가 여러 도구를 자동 조합** |
| 자동화 | 불가능 | **n8n/Zapier로 주간 보고서 자동 발송 가능** |
**결론:**
MCP는 데이터를 전달하는 것이 아니라 **AI와 연결하는 방식**

## 장표 15
**목차:**
04 무엇이 바뀌는가
**메인:**
시간이 가장 많이 쓰이던 구간이 사라진다
**서브:**
같은 데이터라도 사용자에 따라 완전히 다른 도구가 됩니다.
**콘텐츠:**
| 기존 | MCP 이후 |
|------|---------|
| 수집/정리에 전체 시간의 60~70% | AI가 자동 처리, 사용자는 인사이트에 집중 |
| 도구 4개를 왔다갔다 | AI 플랫폼 하나 |
| 경쟁앱 5개 = 같은 조작 5번 반복 | "경쟁앱 5개 비교해줘" 한 마디 |

같은 데이터, 사용자에 따라 다른 도구:
- 리서치/대행사 → 시장 분석 도구
- 앱 마케터 → 경쟁사 인텔리전스
- 경영진 → 보고서 자동화
- 연구원 → 데이터 수집 어시스턴트
**결론:**
사용자의 시간은 **작업에서 판단으로 이동**

## 장표 17
**목차:**
04 무엇이 바뀌는가
**메인:**
MCP는 데이터 파이프라인이지, 결과물을 만드는 엔진이 아니다
**서브:**
MCP가 하는 일은 AI에게 데이터를 가져다주는 것까지입니다. 제안서 생성, 인사이트 도출, 차트 그리기는 전부 AI가 하는 일입니다.
**콘텐츠:**
| MCP가 할 수 있는 것 | MCP가 할 수 없는 것 |
|---|---|
| 데이터를 AI에게 전달 | 제안서/보고서를 직접 생성 |
| 여러 API를 묶어 복합 도구 제공 | 차트를 그려서 반환 |
| 응답 포맷 최적화 | AI의 답변 스타일 제어 |
| 프롬프트 템플릿으로 시나리오 가이드 | 사용자 UI/UX를 직접 제어 |

| 단계 | 역할 |
|---|---|
| MCP (Phase 1) | 데이터를 AI에 연결, 사용 패턴 수집, 시장 반응 검증 |
| Custom GPT / Claude 프로젝트 (Phase 1~2) | MCP 위에 가이드 UI를 얹어 진입 장벽 추가 제거 |
| 자체 AI (Phase 3) | 모바일인덱스 안에서 완전히 제어된 AI 경험 |
**결론:**
MCP는 가장 간단한 형태의 LLM 연동이며, **최종 목표는 모바일인덱스 안에 AI를 만들어 데이터 제공 패러다임을 바꾸는 것**
